# Team Role For Instructor Review

작성일: `2026-04-15`

## 문서 목적

- 2인 팀 기준으로 실제 역할 분담을 강사님께 명확하게 설명하기 위한 문서
- 현재 저장소와 구현 상태가 왜 `After` 중심으로 보이는지 배경까지 함께 정리

## 1. 팀 구성

우리 팀은 총 2명으로 구성되어 있습니다.

- 김민수: `Before` 단계 전담
- 김종원: `After` 단계 전담

역할은 기능 단계 기준으로 나누고 있으며, 현재 개발도 `After`를 먼저 완성하는 방향으로 진행되었습니다.

## 2. 현재 역할 분담

| 담당자 | 담당 범위 | 현재 상태 |
|---|---|---|
| 본인 | `After` 단계 전체, 데이터 전처리 및 split 작업, 현재 저장소의 구현 코드 | 진행 및 구현 완료 범위 다수 존재 |
| 팀원 | `Before` 단계 전체 | 별도 진행 예정 / 담당 범위 고정 |

## 3. 본인 담당 역할

본인은 현재 프로젝트에서 아래 영역을 맡고 있습니다.

### `After` 단계 전체

- 사용자의 분쟁/사후대응 상황 입력 처리
- 관련 조문 retrieval
- grounded answer generation
- citation grounding 구조
- `After` 시나리오 검증
- `After` 데모 흐름 설계

즉, 현재 이 저장소 안에 있는 핵심 backend 로직과 `After` 관련 구현은 본인이 진행한 작업입니다.

### 데이터 전처리 및 split 작업

- 법령 데이터를 실제 서비스에서 사용할 수 있는 형태로 나누고 정리하는 작업
- corpus 기준선 유지에 필요한 전처리 작업
- 현재 retrieval / answer가 동작할 수 있도록 만드는 기반 데이터 작업

이 부분도 본인이 담당하여 진행했습니다.

### 초기 MVP 앱 진행 가능성

- 앱은 먼저 로직을 완성한 쪽이 초기 MVP를 우선 진행하는 방식으로 가고 있습니다.
- 현재는 `After` 관련 로직이 먼저 구현되어 있기 때문에, 초기 MVP 앱 구성도 본인이 먼저 이어서 진행할 가능성이 높습니다.

## 4. 팀원 담당 역할

팀원은 `Before` 단계를 전담합니다.

주요 범위는 아래와 같습니다.

- 계약서 입력 및 분석 흐름
- 계약 단계 위험 탐지
- `Before` 결과 구조화
- 계약서 기반 데모 시나리오 정리
- 이후 `Bridge` 연결에 필요한 `Before` 결과 준비

즉, 팀원은 계약 전 위험 탐지와 계약서 분석 쪽을 중심으로 맡는 구조입니다.

## 5. 현재 저장소 상태와 역할 반영

현재 이 폴더 안에는 사실상 `After` 관련 코드만 들어 있습니다.

그 이유는 다음과 같습니다.

- 본인이 `After` 단계를 전담하고 있음
- 데이터 전처리와 backend 로직도 본인이 먼저 진행했음
- 그래서 현재 저장소 기준 구현 결과가 `After` 중심으로 먼저 쌓인 상태임

따라서 지금 저장소만 보면 역할이 한쪽으로 치우쳐 보일 수 있지만, 실제 팀 내 분담은

- 본인: `After` + 데이터 전처리 + 현재 MVP 구현 선행
- 팀원: `Before`

로 명확히 나뉘어 있습니다.

## 6. 협업 방식

- 본인은 `After` 흐름과 데이터/백엔드 기반을 먼저 완성
- 팀원은 `Before` 흐름을 별도로 정리 및 구현
- 이후 발표 단계에서는 `Before -> Bridge -> After` 전체 흐름으로 통합

현재는 구현 순서상 `After`가 먼저 구체화된 상태이고, 이후 `Before`가 붙으면서 전체 서비스 구조가 완성되는 방식입니다.

## 7. 강사님께 검토받고 싶은 부분

- 2인 팀 기준에서 현재 역할 분담이 적절한지
- `After`를 먼저 구현하고 `Before`를 후속 연결하는 진행 방식이 괜찮은지
- 데이터 전처리와 backend 로직을 한 사람이 맡고, 다른 한 사람이 `Before`를 맡는 구조가 현실적인지
- 발표 때 현재 저장소가 `After` 중심으로 보이는 점을 어떻게 설명하면 가장 자연스러운지

## 참고 메모

- 현재 저장소의 구현 코드는 `After` 중심으로 형성되어 있음
- 데이터 split 및 전처리 작업은 본인이 진행함
- 초기 MVP 앱도 로직을 먼저 완성한 쪽이 우선 진행하는 방식으로 보고 있음
- 최종 발표에서는 팀원의 `Before` 흐름과 본인의 `After` 흐름을 하나의 서비스 흐름으로 묶어 설명할 계획임
