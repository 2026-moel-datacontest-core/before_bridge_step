# 스타팅 후 자체 테스트

## 목적

이 문서는 `quick_start.md` 또는 `scripts/starting.sh`로 서버를 띄운 뒤, 현재 프로젝트가 정상적으로 동작하는지 빠르게 확인하기 위한 자체 테스트 문서다.

이 문서는 아래를 구분하기 위해 사용한다.

- 서버가 단순히 실행 중인지
- 현재 구조가 실제로 정상 동작하는지
- 문제 발생 시 어느 단계에서 깨졌는지

## 테스트 원칙

- 작은 smoke부터 확인한다.
- import, DB, health, build, 실제 흐름 순으로 본다.
- 실패하면 그 단계에서 바로 멈추고 원인을 좁힌다.
- 결과는 다른 채팅으로 넘길 수 있을 정도로 명확하게 기록한다.

## 1. backend import smoke

실행:

```bash
python -c "from backend.main import app; print('import_ok')"
```

정상 기대값:

- `import_ok`

실패 시 확인:

- `backend/.env`
- `DATABASE_URL`
- 최근 migration 상태
- traceback 전체

## 2. PostgreSQL readiness

실행:

```bash
python backend/verify/ensure_postgres_ready.py
```

정상 기대값:

- `ready` 또는 이에 준하는 성공 JSON
- `SELECT 1` probe 성공

실패 시 확인:

- `backend/.env`의 `DATABASE_URL`
- `pg_isready -h 127.0.0.1 -p 5432`
- 필요 시:

```bash
python backend/verify/ensure_postgres_ready.py --start-if-needed
```

## 3. 기본 health 확인

실행:

```bash
curl http://127.0.0.1:8000/health
```

정상 기대값:

- HTTP `200`
- `{"status":"ok"}` 또는 이에 준하는 성공 응답

실패 시 확인:

- backend 프로세스 실행 여부
- `backend/logs/starting_backend.log`

## 4. before health 확인

실행:

```bash
curl http://127.0.0.1:8000/api/v1/before/health
```

정상 기대값:

- `runtime_loaded = true`
- `all_chunks_source = "db"`
- `all_chunks_count = 1724`
- `job_count` 존재

이 단계의 의미:

- `before`가 DB source를 기준으로 startup preload 되었는지 확인
- `before` 캐시가 정상 적재되었는지 확인

실패 시 확인:

- `BEFORE_LAW_SOURCE`
- `sync_before_manual_chunks_to_db.py` 실행 여부
- backend startup 로그

## 5. document draft smoke

실행:

```bash
python backend/verify/check_document_draft.py
```

정상 기대값:

- 모든 fixture case가 `ok ...` 형태로 통과

이 단계의 의미:

- `/api/v1/documents/draft`가 현재 fixture 기준으로 정상 응답하는지 확인

실패 시 확인:

- backend import 상태
- DB readiness
- fixture 변경 여부
- traceback 전체

## 6. frontend build

실행:

```bash
cd frontend
npm run build
cd ..
```

정상 기대값:

- build 성공

이 단계의 의미:

- 현재 프런트 코드가 정적 빌드 가능한 상태인지 확인

실패 시 확인:

- TypeScript 에러
- API 응답 타입 변경 여부
- 최근 `before` / `after` fixture 변경 여부

## 7. frontend 접속 확인

브라우저 또는 `curl`:

```bash
curl -I http://127.0.0.1:5090
```

정상 기대값:

- HTTP `200`

중요:

- 현재 프런트 포트는 `5090`이다.

## 8. 선택 검증: before 실제 업로드 1회

브라우저에서:

- `http://localhost:5090/before`

확인 포인트:

- 업로드 후 job 생성 성공
- 결과까지 polling 완료
- 최신 run이 `backend/data/before_artifacts/runs/` 아래 생성
- DB `before_review_jobs` row가 `completed` 또는 `failed`로 남음

저장 위치:

- [backend/data/before_artifacts/runs](/home/minsoo/after_pipeline/law_main_road/backend/data/before_artifacts/runs)

실패 시 같이 확인할 것:

- 최근 `job_id`
- 최근 run dir
- `/api/v1/before/health`
- `before_review_jobs` 상태

## 9. 선택 검증: after answer / draft 1회

브라우저에서:

- `http://localhost:5090/after`

확인 포인트:

- answer 응답 성공
- intake / draft 흐름 정상
- 최신 run이 `backend/data/after_artifacts/runs/` 아래 생성
- DB `after_artifact_runs`에 answer / draft row가 남음

저장 위치:

- [backend/data/after_artifacts/runs](/home/minsoo/after_pipeline/law_main_road/backend/data/after_artifacts/runs)

실패 시 같이 확인할 것:

- answer 단계인지 draft 단계인지
- 최신 artifact dir
- 관련 backend 로그

## 10. 최소 성공 기준

아래가 모두 만족되면 현재 로컬 상태는 기본적으로 정상이다.

- backend import 성공
- PostgreSQL readiness 성공
- `/health` 성공
- `/api/v1/before/health`에서 `runtime_loaded=true`
- `check_document_draft.py` 성공
- `frontend` build 성공
- frontend `5090` 접속 성공
