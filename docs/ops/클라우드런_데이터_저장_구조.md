# Cloud Run 데이터 저장 구조 가이드

기준일: `2026-04-21`

## 목적

이 문서는 현재 `law_main_road` 프로젝트를 `frontend`와 `backend`로 분리해 Cloud Run에 배포하고, 데이터는 전역 저장소로 분리하려는 상황을 기준으로 권장 저장 구조를 정리한다.

정리 범위:

- 어떤 데이터를 어디에 저장할지
- 왜 그렇게 나누는지
- 현재 `before` / `after` 구조에서 무엇을 유지해야 하는지
- Cloud Run 환경에서 주의해야 할 점
- 단계별 전환 순서

## 전제

모든 전환 작업의 우선 전제는 아래와 같다.

- 현재 사용자에게 보이는 UI 구조와 기존 기능 동작을 유지한다.
- 저장 방식은 바꿀 수 있지만, `before`와 `after`의 결과 해석 방식과 화면 흐름은 갑자기 바꾸지 않는다.
- 특히 `before`는 저장소를 DB로 옮기더라도 현재 결과 생성 규칙을 유지하는 방향으로 전환한다.

## 권장 아키텍처 요약

권장 기본 구조:

- Frontend: `Cloud Run`
- Backend: `Cloud Run`
- 구조화 데이터 / 벡터 검색 / 상태 저장: `Cloud SQL for PostgreSQL + pgvector`
- 파일 원본 / OCR 산출물 / artifact: `Google Cloud Storage`
- 비밀값: `Secret Manager`
- 비동기 분석 작업: `Cloud Tasks` 또는 `Pub/Sub`

핵심 원칙:

- PostgreSQL은 구조화된 데이터와 검색용 코퍼스를 저장한다.
- 파일은 DB가 아니라 Object Storage에 저장한다.
- Cloud Run 로컬 디스크는 임시 작업 공간으로만 사용한다.

## 데이터별 권장 저장소

### 1. 법령 코퍼스

저장소:

- `Cloud SQL for PostgreSQL + pgvector`

포함 대상:

- `law_chunks`
- embedding vector
- citation / metadata
- 검색 인덱스

권장 이유:

- `after`가 이미 이 구조를 기준으로 동작한다.
- `before`도 같은 source of truth를 보게 만들면 데이터 drift를 줄일 수 있다.
- 검색, 필터링, 버전 관리, 운영 검증을 한 지점에서 수행할 수 있다.

주의:

- `before`는 현재 vector retrieval이 아니라 키워드 / citation matching 기반이다.
- `extra` 조항 처리 때문에 전체 법령 코퍼스를 한 번 훑는 성격이 있다.
- 따라서 `before`를 DB로 옮기더라도 retrieval semantics를 `after` 방식으로 바로 바꾸지 않는다.
- 현재 프로젝트 기준으로는 요청마다 일부 조항만 DB에서 부분 조회하는 방식보다, DB를 source of truth로 두고 전체 `law_chunks`를 startup 시 메모리 캐시로 적재하는 방식이 더 적합하다.

### 2. 업로드 원본 파일

저장소:

- `Google Cloud Storage`

포함 대상:

- 계약서 PDF
- 계약서 이미지
- 향후 사용자가 올리는 첨부 파일

권장 이유:

- Cloud Run 로컬 파일시스템은 영속 저장소가 아니다.
- 바이너리를 PostgreSQL에 넣으면 비용과 운영 복잡도가 크게 증가한다.
- GCS는 대용량 파일, 다운로드, 만료 URL, 접근 제어에 적합하다.

주의:

- 현재 로컬의 `backend/data/before_artifacts/runs/...` 구조는 개발용으로는 가능하지만, Cloud Run 배포 구조에서는 영속 저장소로 사용하면 안 된다.
- 공개 정적 URL보다는 signed URL 또는 인증 프록시를 권장한다.

### 3. OCR / 분석 산출물

저장소:

- 원본 산출물 파일: `Google Cloud Storage`
- 구조화된 요약 / 상태 / 참조 메타데이터: `Cloud SQL`

포함 대상:

- `ocr_output.json`
- `review_result.json`
- `user_explanation.md`
- 요약 결과, 상태, 참조 경로

권장 이유:

- 원본 JSON / markdown 결과는 파일로 관리하는 편이 단순하다.
- 서비스 화면에 필요한 핵심 결과값은 DB row로 따로 관리해야 검색, 재조회, 이력 관리가 쉽다.

권장 방식:

- GCS에는 원본 결과 파일 저장
- DB에는 `job_id`, `status`, `gcs_path`, `mime_type`, `summary`, `created_at`, `finished_at` 저장

### 4. before job 상태

저장소:

- `Cloud SQL`

현재 구현 메모:

- 로컬 기준 현재 `before` job 상태는 `before_review_jobs` PostgreSQL 테이블에 저장된다.
- polling API와 `health.job_count`도 DB를 기준으로 동작한다.
- 따라서 기존 `app.state.before_jobs` 메모리 의존은 제거된 상태다.

권장 이유:

- Cloud Run에서는 같은 서비스라도 요청이 다른 인스턴스로 갈 수 있다.
- 메모리 상태 기반 job 관리는 인스턴스가 늘어나는 순간 깨진다.

현재 최소 테이블:

- `before_review_jobs`

향후 확장 후보:

- `before_job_files`
- `before_job_results`

### 5. 설정성 소규모 데이터

저장소:

- 초기 단계: repo 포함 파일 유지
- 운영 변경 필요가 커질 때만 DB 또는 GCS-managed config로 이동

포함 대상:

- `standard_map.json`
- `minimum_wage.yaml`

권장 이유:

- 변경 빈도가 낮고 크기가 작다.
- 처음부터 전부 DB화하면 복잡성만 늘어난다.
- 먼저 법령 코퍼스와 job 상태를 정리하는 것이 우선이다.

## before / after 통합 관점의 권장 방식

### after

`after`는 이미 PostgreSQL + pgvector를 중심으로 설계되어 있으므로, Cloud Run 전환 시에도 큰 방향 변경이 필요 없다.

우선 유지할 점:

- `law_chunks` source of truth 유지
- retrieval / answer / document draft API contract 유지

### before

`before`는 원래 파일 기반 law chunk 로딩과 메모리 job 상태를 사용하던 구조였기 때문에, Cloud Run 전환 전 분리가 필요했다.

우선 변경할 점:

- law chunk source를 DB 기준으로 통합
- artifact 저장을 로컬 디렉터리에서 GCS로 이동
- job 상태를 DB row로 유지하고 artifact 경로와 연결

우선 유지할 점:

- 현재 화면 흐름
- 현재 결과 구조
- 현재 키워드 / citation matching 규칙

중요:

- `before`를 바로 `after`의 vector retrieval 로직으로 바꾸는 것은 권장하지 않는다.
- 첫 단계는 source of truth 통합이지, retrieval semantics 변경이 아니다.

권장 런타임 구조:

- source of truth: PostgreSQL `law_chunks`
- 런타임 처리: startup 시 전체 `law_chunks`를 메모리 캐시로 적재
- 요청 처리: 메모리 캐시 기반으로 기존 `before` 규칙 유지

즉 구조는 아래처럼 나뉜다.

- `after`: DB live retrieval
- `before`: DB source of truth + startup memory cache

## before 캐시 적재 전략

### 핵심 결론

Cloud Run에서는 `before` 캐시를 올릴 수 있다. 다만 정확한 의미는 아래와 같다.

- 배포할 때 서비스 전체에 전역 캐시를 한 번만 올리는 것이 아니다.
- 각 Cloud Run 인스턴스가 시작될 때마다 해당 인스턴스의 메모리에 캐시를 적재하는 방식이다.

즉, 이 캐시는 전역 공유 캐시가 아니라 인스턴스 로컬 메모리 캐시다.

### 왜 startup cache가 필요한가

현재 `before`는 아래 특성이 있다.

- vector retrieval이 아니라 키워드 / citation matching 기반이다.
- extra 조항 처리 때문에 전체 법령 코퍼스를 한 번 훑는 성격이 있다.
- 따라서 매 요청마다 DB에서 일부 조항만 부분 조회하는 구조보다, 전체 코퍼스를 메모리에 적재한 뒤 기존 규칙을 유지하는 쪽이 안전하다.

이 프로젝트에서 권장하는 흐름은 아래와 같다.

1. 법령 원본은 PostgreSQL `law_chunks`에 저장한다.
2. `before` backend 인스턴스가 시작될 때 전체 `law_chunks`를 읽는다.
3. 이를 `list[dict]` 형태의 런타임 캐시로 메모리에 올린다.
4. `before` 단계는 이 캐시를 사용해 지금과 동일하게 비교, 매칭, extra 조항 탐색을 수행한다.

### 배포 트리거와 startup 트리거의 차이

운영 개념상으로는 "배포 후 캐시 준비"처럼 보일 수 있지만, 실제 구현 기준은 인스턴스 startup 트리거다.

배포 트리거 관점:

- 새 revision을 배포하면 Cloud Run이 새 인스턴스를 만들 수 있다.
- 하지만 캐시 적재가 "배포 1회당 정확히 한 번만" 일어난다고 보장할 수는 없다.
- 이후 cold start, scale-out, 인스턴스 재시작 시에도 다시 적재될 수 있다.

startup 트리거 관점:

- 실제로 신뢰할 수 있는 기준은 인스턴스 startup 시점이다.
- 각 인스턴스는 시작될 때 필요한 메모리 캐시를 직접 적재해야 한다.
- 따라서 구현은 FastAPI `startup` 또는 `lifespan` 단계에서 이루어지는 것이 자연스럽다.

정리하면:

- 운영 개념상: 배포 후 캐시 준비
- 구현 개념상: 인스턴스 시작 시 캐시 적재

### 권장 구현 흐름

1단계. DB를 원본으로 사용

- `law_chunks`는 PostgreSQL에만 저장한다.
- `before_assets/law_chunks/all_chunks.json` 의존은 제거하거나 fallback 용도로만 유지한다.

2단계. startup 시 캐시 적재

- FastAPI app startup 단계에서 `law_chunks` 전체를 조회한다.
- DB row를 현재 `before` 로직이 기대하는 dict shape로 변환한다.
- 결과를 `app.state.all_chunks` 같은 메모리 영역에 보관한다.

현재 구현 메모:

- `backend.main`의 parent app `lifespan`에서 `before_app` runtime preload를 명시적으로 호출한다.
- 즉 mounted sub-app의 첫 요청에 기대지 않고, 서비스 startup 시점에 `before` 캐시 적재를 먼저 수행하도록 맞췄다.

3단계. 요청 처리에서는 캐시 사용

- `before` 분석 요청이 들어오면 DB를 다시 스캔하지 않는다.
- 이미 메모리에 올라온 `all_chunks`를 사용한다.

4단계. 캐시 미적재 대비 fallback

- startup 실패 시 서비스 전체를 fail-fast 할지
- 또는 파일 fallback을 둘지
- 이 부분은 운영 정책에 따라 선택한다.

현재 단계에서는 아래 두 가지 중 하나가 현실적이다.

- 운영 엄격 모드: DB 적재 실패 시 앱 startup 실패
- 로컬 실험 모드: `BEFORE_LAW_SOURCE=file|db` 같은 env flag로 분기

### 장점

- `before`의 기존 semantics를 거의 유지할 수 있다.
- extra 조항처럼 전체 법령을 훑는 로직을 그대로 유지할 수 있다.
- source of truth를 `after`와 같은 DB로 통합할 수 있다.
- Cloud Run 구조에서도 파일 로컬 저장소 의존을 줄일 수 있다.

### 단점과 trade-off

1. 인스턴스별 중복 메모리 사용

- scale-out되면 여러 인스턴스가 같은 코퍼스를 각자 메모리에 올린다.

2. cold start 시 로딩 비용

- 인스턴스 시작 시 전체 코퍼스를 읽어야 하므로 startup 시간이 늘어난다.

3. 캐시 영속성 없음

- 캐시는 인스턴스 메모리이므로 재시작 시 사라진다.
- 항상 다시 적재 가능해야 한다.

4. 데이터 갱신 반영 전략 필요

- DB의 `law_chunks`가 바뀌면 기존 인스턴스 메모리 캐시는 자동 반영되지 않는다.
- 새 revision 배포, 인스턴스 재시작, 수동 refresh 전략 중 하나가 필요하다.

## Cloud Run 환경에서 주의할 점

### 1. 로컬 디스크 비영속성

- Cloud Run 컨테이너 안에 생성한 파일은 인스턴스 수명 동안만 유효하다.
- 재시작, scale-out, 새 revision 배포 시 파일이 사라질 수 있다.

결론:

- `before_artifacts/runs` 같은 구조는 개발용 캐시까지만 허용
- 영속 보관 대상은 반드시 GCS 또는 DB로 이동

### 2. 인스턴스 간 메모리 공유 불가

- Cloud Run 인스턴스 A의 메모리는 인스턴스 B와 공유되지 않는다.
- 따라서 job queue, 업로드 상태, 진행률, poll 응답 상태를 메모리에 두면 안 된다.

결론:

- 상태는 DB
- 비동기 실행은 queue 기반

### 3. DB connection 관리

- Cloud Run가 scale-out되면 DB connection 수가 빠르게 증가할 수 있다.
- PostgreSQL pool을 로컬 개발 기준으로 크게 잡으면 운영에서 쉽게 문제가 난다.

권장:

- 보수적인 pool size
- Cloud SQL connector 또는 적절한 connection management 적용

### 4. artifact 접근 제어

- 민감 문서를 공개 정적 URL로 노출하면 안 된다.
- 계약서 원본, OCR 결과, 리뷰 파일은 signed URL 또는 인증된 다운로드 경로가 적절하다.

### 5. 장기 작업 처리

- OCR / 분석이 길어지면 단일 HTTP 요청으로 끝내기 어렵다.
- Cloud Run request timeout, 재시도, 중복 실행을 고려해야 한다.

권장:

- 요청은 job 생성까지만 처리
- 실제 분석은 Cloud Tasks 또는 Pub/Sub + worker 흐름으로 분리

### 6. before 캐시는 전역 공유가 아니다

- Cloud Run 인스턴스 간 메모리는 공유되지 않는다.
- 인스턴스 A가 적재한 `before` 캐시는 인스턴스 B가 볼 수 없다.

결론:

- 캐시는 "있으면 빠른 것"으로 봐야 한다.
- 항상 startup 시 재생성 가능해야 한다.

### 7. min instances와 startup 비용

- `min instances`를 주면 warm 인스턴스를 유지하는 데 도움이 된다.
- 하지만 인스턴스가 영원히 유지되는 것은 아니다.
- startup에서 너무 많은 작업을 하면 cold start 체감이 커진다.

권장:

- `before` backend는 `min instances=1` 이상을 검토
- cold start가 체감되면 `startup CPU boost` 검토
- 캐시 적재 위치는 FastAPI `startup` 또는 `lifespan`

현재 구현 메모:

- 로컬 기준으로는 parent app `lifespan`에서 `before` preload를 수행하고, `before /health` 첫 응답에서 `runtime_loaded=true`, `all_chunks_source=db`, `all_chunks_count=1724`를 확인했다.

## 권장 전환 순서

현재 UI와 기능을 유지하려면 아래 순서가 가장 안전하다.

### 1단계. 법령 source of truth 통합

- `before`가 별도 JSON 복사본이 아니라 `after`와 같은 `law_chunks` DB를 보게 만든다.
- 단, `before` 결과 생성 로직은 유지한다.

### 2단계. before artifact 저장소 전환

- 업로드 원본과 결과 파일을 로컬 디스크에서 GCS로 옮긴다.
- DB에는 경로와 상태만 저장한다.

### 3단계. before job 상태 DB화

- 완료 상태:
- `app.state.before_jobs` 제거 완료
- polling API는 `before_review_jobs`를 조회하도록 전환 완료
- `health.job_count`도 DB row count를 사용한다

### 4단계. 비동기 실행 계층 추가

- 분석이 무거워지면 Cloud Tasks 또는 Pub/Sub를 붙인다.
- backend API는 동기 요청이 아니라 job orchestration 역할로 줄인다.

### 5단계. 운영 보안 정리

- Secret Manager 전환
- signed URL 또는 인증 프록시 적용
- 서비스 계정과 최소 권한 IAM 정리

## 추천 로컬 실험 방식

Cloud Run 전환 전에 로컬에서 아래처럼 먼저 검증하는 것이 좋다.

1. `law_chunks`는 PostgreSQL에 적재
2. `before`는 startup 시 DB에서 전체 읽기
3. 이를 메모리 캐시로 적재
4. 기존 파일 기반 결과와 비교

로컬에서 확인할 비교 포인트:

- 일반 계약서 검토 결과 동일 여부
- extra 조항 매핑 동일 여부
- `citation_label` 매칭 동일 여부
- startup 시간 증가 정도

## 추천 최소 스키마 방향

## 결론

현재 프로젝트를 Cloud Run 구조로 옮길 때 가장 좋은 방향은 아래와 같다.

- 법령 / 메타데이터 / 상태: `Cloud SQL for PostgreSQL + pgvector`
- 업로드 파일 / OCR 결과 / artifact: `Google Cloud Storage`
- 비동기 처리: `Cloud Tasks` 또는 `Pub/Sub`
- secret: `Secret Manager`

그리고 가장 중요한 구현 원칙은 아래 한 줄로 정리할 수 있다.

`before`와 `after`의 source of truth는 통합하되, 현재 사용자에게 보이는 UI와 기능 결과는 유지한다.

`before`는 DB를 원본 저장소로 사용하되, 실행 시점에는 전체 법령 snapshot을 메모리에 적재해 기존 처리 규칙을 유지한다.
