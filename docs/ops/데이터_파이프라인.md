# 데이터 파이프라인

## 목적

이 문서는 `law_main_road`의 데이터가 어디서 와서 어떤 단계를 거쳐 현재 서비스가 사용하는 상태가 되는지 정리한 문서다.

특히 이 문서는 아래 2가지를 동시에 설명한다.

- 현재 로컬 개발 기준 데이터 파이프라인
- 이후 Cloud Run / Cloud SQL / GCS 마이그레이션에서 민감한 지점

즉, 단순히 전처리 스크립트 순서만 적는 문서가 아니라, 현재 source of truth와 운영 전환 시 주의할 점까지 함께 보는 문서다.

## 전제

- 현재 사용자에게 보이는 UI 구조와 기존 기능 동작은 유지한다.
- 데이터 저장 위치는 바꿀 수 있어도 `before`와 `after`의 결과 해석 규칙은 갑자기 바꾸지 않는다.
- 현재 source of truth는 `backend/data/law_chunks/all_chunks.json`이다.

## 데이터 파이프라인 전체 지도

현재 프로젝트의 데이터 흐름은 크게 4단계다.

1. 법령 원천 데이터 수집
2. chunk 전처리 파이프라인
3. PostgreSQL + pgvector 적재
4. 서비스 런타임 사용

이를 현재 기준으로 적으면 아래와 같다.

```text
data/legalize-kr
  -> scripts/step1,4,5,6,7,8,9,10
  -> backend/data/law_chunks/all_chunks.json
  -> backend/scripts/ingest_chunks.py
  -> PostgreSQL law_chunks
  -> backend/scripts/embed_chunks.py
  -> pgvector embedding + HNSW index
  -> after: live retrieval
  -> before: DB source + startup cache
```

## 7. 런타임 사용 방식

### after

`after`는 DB live retrieval을 사용한다.

흐름:

- 사용자 질의
- query embedding
- `law_chunks` vector retrieval
- answer generation
- document draft generation

즉 `after`는 DB를 직접 조회하는 구조다.

### before

`before`는 현재 DB source + startup cache 구조다.

흐름:

- source of truth는 DB `law_chunks`
- backend startup 시 전체 `law_chunks`를 메모리 캐시로 적재
- 요청 처리 시 이 캐시를 사용해 키워드 / citation / extra 조항 탐색 수행

중요:

- `before`는 현재 vector retrieval이 아니다.
- `after`와 같은 retrieval semantics로 바로 통합하지 않는다.
- 현재 규칙을 유지한 채 source of truth만 DB로 통합한 상태다.

## 9. Cloud 마이그레이션에서 특히 신경 써야 하는 부분

이 문서에서 가장 중요한 운영 포인트는 여기다.

### 1. source of truth를 흔들지 말 것

클라우드로 옮겨도 아래는 유지해야 한다.

- 법령 source of truth: `all_chunks.json` -> `law_chunks`
- `before` supplement sync
- `before`의 retrieval semantics
- `after`의 DB live retrieval

즉, 클라우드 전환은 저장 위치와 런타임 환경을 바꾸는 것이지, 법령 의미 해석 규칙을 바꾸는 작업이 아니다.

### 2. Cloud Run 로컬 디스크를 저장소로 쓰지 말 것

현재 로컬에서는 `before_artifacts`, `after_artifacts`를 디스크에 저장하지만, Cloud Run에서는 이걸 영속 저장소처럼 쓰면 안 된다.

향후 전환 방향:

- 로컬 artifact 디렉터리 -> `GCS`
- DB 메타데이터는 그대로 유지 또는 확장

### 3. before는 DB 부분 조회보다 startup cache가 우선

`before`는 extra 조항 처리 때문에 전체 코퍼스를 훑는 성격이 있다.

그래서 클라우드에서도 권장 방식은 같다.

- source of truth: Cloud SQL
- 런타임 처리: startup cache

주의:

- 이 캐시는 Cloud Run 서비스 전역 캐시가 아니라 인스턴스별 메모리 캐시다.
- scale-out, cold start, 재시작 시 다시 적재될 수 있다.

### 4. embedding / ingestion을 애플리케이션 startup에 넣지 말 것

클라우드 마이그레이션에서 자주 생기는 실수는 아래다.

- 앱이 뜰 때마다 ingestion 수행
- 앱이 뜰 때마다 embedding 재생성

이건 금지하는 게 맞다.

권장:

- chunking / ingestion / embedding은 batch 또는 별도 운영 절차
- 서비스 startup은 readiness와 cache preload까지만 담당

### 5. DB migration과 데이터 적재를 분리할 것

운영 전환 시 아래 순서를 권장한다.

1. DB schema migration
2. `law_chunks` 적재
3. embedding 채우기
4. before supplement sync
5. backend 배포
6. frontend 배포

즉 alembic과 corpus 적재를 한 번에 섞지 않는 게 안전하다.

### 6. 검증 기준을 명확히 유지할 것

클라우드로 옮길 때도 아래 기준은 그대로 써야 한다.
