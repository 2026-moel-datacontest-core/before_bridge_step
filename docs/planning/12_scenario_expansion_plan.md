## 현재 판단 기준

시나리오는 먼저 아래 둘 중 하나로 분류한다.

### A. Current Corpus Covered

- 핵심 조문이 이미 `backend/data/law_chunks/all_chunks.json`에 존재
- 현재 retrieval + grounded answer 구조로 데모 가능한 범위

- 핵심 조문이 현재 corpus에 없음
- 또는 조문은 있으나 데모 목표를 충족하기 어려울 정도로 관련 맥락이 부족함
- 이 경우 다음 순서 검토:
  1. source of truth 범위 변경
  2. chunking 재실행
  3. ingestion 재실행
  4. embedding 재생성
  5. retrieval / answer eval 재확인

### 유형 정의

- `Full`: Before -> Bridge -> After가 한 흐름으로 이어지는 대표 시연 시나리오
- `Before`: 계약 스캔 / 사전 위험 탐지 / 계약 단계 보호 중심 시나리오
- `After`: 실제 분쟁 발생 후 권리 진단 / 대응 / 구제 중심 시나리오

현재 기준으로는 `Full`, `Before`, `After`를 모두 이 문서에 누적 관리하는 방식을 기본으로 한다.
시나리오 수가 많아지기 전까지는 한 문서에 모아두는 편이 다음 세션에서 데이터 확장 필요 여부와 데모 우선순위를 함께 판단하기 쉽다.

## SCN-001

### 이름

외국인 특화 Full 시연: 자체 양식 계약서 -> 기숙사 꼼수 및 차별 -> 사업장 변경 지원

### 통합 시나리오 요약

- Before:
  - 외국인 근로자가 제조업 입사 직전 회사가 제시한 자체 양식 근로계약서를 앱으로 스캔
  - 임금과 근로시간은 대략 적혀 있으나, 기숙사 위치, 면적, 시설, 공제 비용은 서면 명시 없이 구두로만 설명
  - 외국인 고용허가제 기준의 표준근로계약서 사용 여부도 불명확한 상태
- Bridge:
  - AI가 자체 양식 계약서를 분석하고 사전 위험을 예측
  - 표준근로계약서 미사용 가능성 경고
  - 기숙사 정보 누락 및 과도한 숙소비 공제 위험 경고
  - 근로기준법 제17조상 필수 근로조건 서면 명시 부족 경고
- After:
  - 실제 근무 후 월급에서 과도한 기숙사비가 공제됨
  - 실제 숙소는 비닐하우스 형태의 임시 가건물로 매우 열악함
  - 강제 초과근무, 폭언, 차별 대우까지 겹침
  - 근로자가 앱에 "회사 잘못으로 사업장을 변경하고 싶은데 가능한가요?"라고 질문하는 상황으로 이어짐

### 데모 포인트

- 외국인 근로자 특화라는 프로젝트 정체성이 가장 잘 드러남
- Before -> Bridge -> After 흐름이 자연스럽게 이어짐
- 계약 스캔 기반 사전 위험 예측과 사후 권리 구제까지 한 흐름으로 연결 가능
- 단일 시연 안에서 근로조건 명시, 표준계약, 기숙사, 차별, 사업장 변경, 증거 수집, 사유서 작성까지 확장 가능

### AI 스캔 결과 (Bridge)

- 표준근로계약서 미사용 경고:
  - 외국인 고용허가제(E-9 등) 기준의 표준근로계약서 양식이 아닐 수 있으며, 필수 보호 조항 누락 위험이 큼
- 기숙사 정보 누락 적발:
  - 기숙사 위치, 면적, 시설 조건, 비용 공제 내역이 서면으로 명시되지 않아 추후 과도한 숙소비가 일방 공제될 위험이 큼
- 근로기준법 위반 경고:
  - 임금, 근로시간, 휴일 등 필수 근로조건이 서면으로 충분히 명시되지 않았을 가능성이 있어 `근로기준법 제17조` 점검 필요

### AI 사후 대응 (After Solution)

- 권리 진단:
  - 열악한 기숙사 제공, 약정하지 않은 과도한 임금 공제, 차별 또는 폭언 정황이 있으면 사용자 귀책사유 판단과 사업장 변경 허용 가능성을 검토할 수 있음
- 맞춤형 증거 수집 가이드:
  - 기숙사 내외부 사진
  - 과도 공제가 반영된 급여명세서 또는 통장 입금 내역
  - 차별 또는 폭언 관련 메모, 메시지, 녹취 가능 여부 안내
- 후속 작성 지원:
  - 사업장 변경 사유서 초안 자동 작성 기능으로 연결 가능한 대표 시나리오
  - 한국어와 모국어 병기 지원은 향후 데모 확장 포인트로 유지

- 이번 SCN-001 기준으로는 source update 불필요
- chunking / ingestion / embedding 재실행도 불필요
- `backend/data/law_chunks/` 산출물과 `all_chunks.json`은 이번 턴에 수정하지 않음
- 우선순위는 데이터 보강이 아니라 복합 질의에서의 retrieval 랭킹 안정화 확인이다

### 다음 세션 권장 실행 순서

1. 이 시나리오를 기준으로 단계별 demo query를 분리 작성
   - Before 스캔 경고 질의
   - After 권리 진단 질의
   - 필요 시 사유서 작성 연결 질의
2. 현재 retrieval 결과가 필요한 조문을 안정적으로 함께 끌어오는지 확인
3. answer route가 `근로기준법 제17조`, `외국인고용법 제9조`, `제22조`, `제25조`, `시행규칙 제16조`를 적절히 grounding하는지 확인
4. 특히 `제22조`의 차별 금지와 기숙사 제공 문맥이 함께 surface되는지 확인
5. 만약 특정 논점이 retrieval에서 빠지면:
   - query normalization 보강
   - clause ranking / adjacent context expansion 보강
6. 현재 corpus로도 데모가 부족하다고 확인될 때만 데이터 확장 여부를 다시 판단

### 리스크 / 메모

- `외국인근로자의 고용 등에 관한 법률 제22조`는 현재 corpus에서 `차별 금지`와 `기숙사의 제공 등`이 각각 별도 label로 잡혀 있다.
- 실제 데모 질의에서는 차별과 기숙사 이슈가 동시에 들어가므로, retrieval이 두 문맥을 함께 끌어오는지 확인이 필요하다.
- source 원문과 chunk 본문에서는 `기숙사의 제공 등`이 `제22조`의 중복 조문으로 관리되지만, `제25조`와 관련 시행령 문구 안에는 내부 참조가 `제22조의2`로 나타난다. 따라서 데모 문안에서는 source/corpus의 실제 citation label을 우선 기준으로 삼는 편이 안전하다.
- 사업장 변경 사유서 자동 생성은 현재 answer route 범위를 넘는 후속 기능이므로, 문서상 데모 확장 포인트로만 유지하고 실제 구현 범위와 혼동하지 않도록 주의가 필요하다.
- 현재 병목은 데이터 부재보다 retrieval-grounding-answer 연결 품질, 특히 Full 단일 질의에서의 랭킹 안정성일 가능성이 더 높다.

### 데모 스크립트 메모

- Before / Bridge:
  - 외국인 근로자가 자체 양식 계약서를 스캔하면 AI가 표준근로계약서 미사용 가능성, 기숙사 정보 누락, `근로기준법 제17조`상 서면 명시 부족 위험을 경고
- After:
  - 몇 달 뒤 실제로 열악한 숙소, 과도 공제, 차별과 폭언이 발생하면 AI가 사용자 귀책사유 여부와 사업장 변경 가능성을 법 조문 근거와 함께 설명
- Expansion:
  - 이후 증거 수집 체크리스트와 사업장 변경 사유서 작성 지원으로 이어지는 흐름을 데모 내 후속 기능 포인트로 제시 가능

## SCN-002

### 유형

`Before`

### 이름

알바/생활밀착형: 최저임금 미달 및 수습 꼼수 적발

### 시나리오 요약

- 배경:
  - 아르바이트생 또는 단기 신입사원이 입사 전 근로계약서를 앱으로 스캔
  - 계약서상 시급란에 당해 연도 최저임금보다 낮은 금액이 적혀 있거나, 특약사항에 `수습기간 3개월간은 최저임금의 90%만 지급` 같은 문구가 포함된 상태
- 핵심 위험:
  - 최저임금 미달 약정
  - 수습 감액 예외 조항 오남용
  - 장기 근속 시 낮은 기본급이 유지되면 퇴직금 산정에도 불이익 가능

### Bridge 위험 태그

- `최저임금법 위반`
- `수습 예외 조항 오남용`
- `퇴직금 산정 불이익`

### 데모 포인트

- 매우 생활밀착형이고 직관적이라 관객 이해가 빠름
- 단순 숫자 비교를 넘어, 계약기간과 직무 성격을 함께 해석하는 AI 판단을 보여주기 좋음
- “서명 전 위험 경고”라는 Before 전용 UX를 가장 선명하게 보여줄 수 있음

### 사용자 질문 예시

> 편의점 알바 계약서를 찍어보니 계약기간은 6개월이고, 수습 3개월 동안은 시급 90%만 준다고 적혀 있어요. 시급도 올해 최저임금보다 낮아 보이는데, 이런 계약은 그대로 서명해도 되나요?

### AI 스캔 결과 목표

- 최저임금 위반 즉시 탐지:
  - 스캔된 시급 숫자를 해당 연도 최저임금과 비교해 경고
- 수습 감액 예외 오남용 진단:
  - 계약기간이 1년 미만인지
  - 직종이 단순노무에 가까운지
  - `수습 3개월 90%` 조항이 유효한 예외인지 또는 무효인지 설명
- 선제적 권리 안내:
  - 서명 전 정정 요청
  - 익명 상담 또는 신고 전 단계 안내
  - 장기적으로 퇴직금 계산에도 불리할 수 있다는 점 설명

### 핵심 조문

- `최저임금법 제5조 (최저임금액)`
- `최저임금법 시행령 제3조 (수습 중에 있는 근로자에 대한 최저임금액)`
- `최저임금법 제6조 (최저임금의 효력)`

### 보조 근거

- `근로기준법 제15조 (이 법을 위반한 근로계약)`
- `근로기준법 제17조 (근로조건의 명시)`

### 현재 corpus 커버 판정

판정: `Partial`

- `법률 설명형` 질의는 `Current Corpus Covered`
- `시급 자동 판정형` 데모는 `Corpus Expansion Needed`

이번 세션 재검증 기준:

- 원문: `data/legalize-kr/kr/최저임금법/{법률.md,시행령.md,시행규칙(고용노동부령).md,시행규칙.md}`
- source of truth: `backend/data/law_chunks/all_chunks.json`
- PostgreSQL `law_chunks`

현재 `backend/data/law_chunks/all_chunks.json`과 PostgreSQL `law_chunks`에서 확인된 관련 citation label:

- `최저임금법 제5조 (최저임금액)`
- `최저임금법 시행령 제3조 (수습 중에 있는 근로자에 대한 최저임금액)`
- `최저임금법 제6조 (최저임금의 효력)`

법 조문 자체는 현재 corpus에 들어 있다. 확인된 핵심 조건은 아래와 같다.

- 수습 감액 예외는 `1년 이상의 기간을 정하여 근로계약을 체결`한 경우여야 함
- `수습을 시작한 날부터 3개월 이내`여야 함
- `단순노무업무` 종사자는 예외에서 제외됨
- `시행령 제3조`에는 수습 감액 폭이 `100분의 10`으로 명시됨
- `제6조`에는 최저임금 미달 약정 무효와 최저임금 이상 지급 의무가 포함됨

즉 `계약기간 6개월 + 수습 3개월 90%` 유형의 위법성 판단 근거는 현재 corpus 안에 상당 부분 존재한다.

다만 demo에서 약속한 수준의 자동 판정까지 가려면 현재 corpus만으로는 부족한 부분이 있다.

- 해당 연도 `실제 시간급 최저임금액` 고시 본문은 현재 source 범위에 없음
- `단순노무업무로 고용노동부장관이 정하여 고시한 직종`의 실제 목록 또는 직종 매핑 근거도 현재 source 범위에 없음
- 따라서 “편의점 / 카페 / 식당 알바는 수습 감액 절대 불가” 같은 강한 자동 판정은 별도 rule layer 또는 추가 데이터가 필요할 수 있음

### 현재 단계 권장 결론

- `수습 감액 예외 오남용` 설명형 데모는 현재 corpus로 가능
- `시급 자동 비교`까지 포함한 완성형 데모는 현재 corpus만으로는 부족하다
- 최종 커버 판정은 `Partial`로 유지하되, 자동 판정형 범위만 보면 `Corpus Expansion Needed`로 본다

### 데이터 조치 판단

- 이번 세션에서는 `backend/data/law_chunks/`와 `all_chunks.json`을 수정하지 않았다.
- 이유:
  - 필요한 추가 근거가 현재 `data/legalize-kr/kr/최저임금법` source 범위에 존재하지 않는다.
  - 원문 디렉토리에는 법률/시행령/시행규칙만 있고, 연도별 최저임금 고시나 단순노무 직종 고시는 확인되지 않았다.
- 따라서 ingestion / embedding 재실행도 하지 않았다.

### query 검증

1. `편의점 알바 계약서를 찍어보니 계약기간은 6개월이고, 수습 3개월 동안은 시급 90%만 준다고 적혀 있어요. 이런 계약은 그대로 서명해도 되나요?`
   - retrieval `top_k=10`의 top 3가 `시행령 제3조`, `법 제5조`, `법 제6조`로 고정 확인됨
   - answer cited_articles도 위 3개를 포함했고 grounding 위반은 없었음
2. `근로계약서 시급이 올해 최저임금보다 낮고, 수습 3개월 동안 90%만 지급한다고 적혀 있습니다. 계약기간은 6개월인데 위법 가능성이 있나요?`
   - retrieval top 3가 동일하게 유지됨
   - answer cited_articles는 `시행령 제3조`, `법 제5조`, `법 제6조`, `근로기준법 제15조`, `최저임금법 제28조`였고 grounding 위반은 없었음
3. `근로계약서에 시급 9500원, 계약기간 6개월, 수습 3개월 동안은 시급 90%만 준다고 적혀 있어요. 올해 기준으로 문제가 있나요?`
   - retrieval은 동일하게 핵심 조문을 회수했지만, answer는 현재 연도의 실제 최저임금액 숫자를 surface하지 못했음
   - numeric 비교형 데모에는 source gap이 남아 있음

### 다음 세션 권장 실행 순서

1. 이 시나리오는 `법률 설명형 Before 데모`로 범위를 고정
2. `계약기간 6개월 + 수습 3개월 90%` 문맥에서 감액 조항 무효 가능성을 설명하는 흐름만 데모에 사용
3. 실제 연도별 최저임금 숫자 비교나 단순노무 직종 자동 판정은 현재 데모 범위에서 제외
4. 따라서 현 시점에서는 추가 source / ingestion / embedding 작업 없이 유지

### 리스크 / 메모

- 초안의 `근로자퇴직급여 보장법 제9조`는 퇴직금 지급 시기 조문이라 이번 시나리오의 핵심 조문으로 보기 어렵다.
- 실제 corpus citation label은 `근로자퇴직급여 보장법`처럼 법령명에 공백이 들어가므로, 초안 표기와도 정확히 일치하지 않았다.
- 현재 corpus 기준으로는 `최저임금 위반 + 수습 예외 오남용`이 중심 논점이고, `퇴직금 불이익`은 보조 메시지로 다루는 편이 안전하다.

### 데모 스크립트 메모

- Before:
  - 편의점 또는 카페 아르바이트 계약서를 스캔하면 AI가 시급, 계약기간, 수습 조항을 동시에 읽는다.

  - AI가 `계약기간 6개월`, `수습 3개월`이라는 문맥을 함께 봐서 감액 조항 무효 가능성을 설명하는 데는 현재 corpus가 충분하다.
- Caution:
  - 실제 연도별 최저임금액 비교와 단순노무 직종 분류는 현재 source/corpus에 직접 근거가 없으므로, 이번 기준선에서는 데모 범위에서 제외하는 편이 안전하다.

## SCN-003

### 유형

`Before`

### 이름

장애인 특화: 일반 계약서 분석 및 맞춤형 지원 제도 안내

### 시나리오 요약

- 배경:
  - 장애인 근로자가 입사 전 제공받은 일반 표준근로계약서를 앱으로 촬영
  - 앱 설정상 사용자는 자신의 장애 유형을 미리 입력해 둔 상태
  - 계약서 자체에 명백한 불법 조항은 없지만, 장애 특성을 고려한 조치나 근무 환경 관련 내용은 전혀 적혀 있지 않음
- 핵심 포인트:
  - 불법 조항 탐지보다 `숨겨진 권리와 지원 제도 안내`가 중심
  - 정당한 편의제공 의무를 사용자에게 선제적으로 인지시킴
  - 보조공학기기, 근로지원인, 관련 기관 연계 같은 지원 중심 UX를 보여주기 좋음

### Bridge 위험 태그

- `정당한 편의제공 의무 누락`
- `정보 사각지대`
- `부당 직무 배치 위험`

- “독소조항 탐지”가 아니라 “권리·혜택 발견”을 보여주는 긍정형 Before 시나리오

- 계약서가 법적으로 깨끗해 보여도 정보 비대칭이 남을 수 있다는 점을 잘 드러냄

### AI 스캔 결과 목표

- 기본 위험 점검 완료:
  - 임금, 근로시간, 휴일 같은 기본 기재사항은 이상 없음을 먼저 확인

- 맞춤형 지원 제도 안내:
  - 보조공학기기 지원

  - 단순히 “안 된다/된다”를 넘어서, 재신청을 어떻게 남기고 무엇을 증거로 확보해야 하는지까지 단계별로 안내한다.
- Expansion:
  - 이후 법적 근거가 포함된 공식 재신청서 또는 내용증명 초안 자동 작성 기능으로 자연스럽게 확장 가능하다.

## 다음 시나리오 수집 형식

추가 시나리오는 아래 형식으로 누적한다.

```text
[SCN-00X]
- 유형: Full / Before / After
- 이름:
- 사용자 질문/입력 예시:
- 보여주고 싶은 데모 포인트:
- 기대 답변 요약:
- 꼭 들어가야 하는 법/조문:
- 현재 corpus 안일 것 같은지 추정:
- 우선순위: must / should / nice
```

다음 시나리오를 받으면 이 문서에 이어서 추가하고, 각 시나리오별로 `Current Corpus Covered` 또는 `Corpus Expansion Needed` 판정을 붙인다.

권장 누적 순서:

1. 대표 `Full` 시나리오를 먼저 고정
2. `Before` 전용 시나리오 2개를 이어서 추가
3. `After` 전용 시나리오 2개를 이어서 추가
4. 각 시나리오마다
   - 현재 corpus 커버 여부
   - 데이터 추가 필요 여부
   - 데모 우선순위
   를 함께 기록
