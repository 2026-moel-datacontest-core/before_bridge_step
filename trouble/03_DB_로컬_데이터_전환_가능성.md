# DB To Local Data Feasibility

기준일: `2026-04-21`

공통 전제:

- 모든 수정은 현재 사용자에게 보이는 UI 구조와 기존 기능 동작을 유지하는 범위에서만 진행한다.
- 정리, 보안 보강, 안정화, 리팩터링이 필요하더라도 현재 확인 가능한 화면 흐름과 기능 결과를 바꾸지 않는 것을 우선한다.

목적:

- 현재 merged 상태에서 `before` / `after` 파이프라인이 DB 없이 로컬 파일만으로 동작 가능한지 확인한다.
- 특히 `before_web/assets/data/*` 와 `backend/data/law_chunks/all_chunks.json` 를 기준으로 대체 가능 범위를 정리한다.

---

## 1. 결론 요약

### before

가능하다.

- 원본 `before_web`는 애초에 DB 없이 로컬 파일 기반 구조다.
- 실제로 사용하는 파일:
  - `assets/law_chunks/all_chunks.json`
  - `assets/data/standard_map.json`
  - `assets/config/minimum_wage.yaml`

즉 `before`는 원본 backend 를 다시 살릴 때도 DB가 필수는 아니다.

### after

현재 구조 그대로는 불가능하다.

이유:

- `backend/app/services/retrieval.py` 가 `SessionLocal` + `LawChunk.embedding` + pgvector 검색을 직접 사용한다.
- 현재 `backend/data/law_chunks/all_chunks.json` 에는 **본문 메타데이터와 `embedding_text`만 있고, 임베딩 벡터 값은 없다.**
- 따라서 DB를 제거하면 지금의 벡터 검색 방식은 그대로 유지할 수 없다.

하지만:

- **기능 축소 또는 검색 방식 변경을 허용하면 local 파일 기반으로 바꾸는 것은 가능하다.**

---

## 2. before 현재 코드 기준

원본 before_web 의 로컬 파일 의존 경로:

- `domain/domain/core/settings.py`
- `api-service/app/api/app.py`
- `domain/domain/contract_analysis/law_retriever.py`
- `domain/domain/contract_analysis/section_comparator.py`
- `domain/domain/contract_analysis/rule_validator.py`

사용 파일:

- `assets/law_chunks/all_chunks.json`
- `assets/data/standard_map.json`
- `assets/config/minimum_wage.yaml`

의미:

- before 쪽은 DB를 전혀 전제하지 않는다.
- 계약서 비교 / 규칙 검증 / 추가 법령 조회는 전부 로컬 파일 기반이다.

즉:

- before 실제 API 를 나중에 붙일 때도 “DB 이전 없이 로컬 파일 기반 복원”이 가능하다.

---

## 3. after 현재 코드 기준

핵심 경로:

- `backend/app/services/retrieval.py`
- `backend/app/models/law_chunk.py`
- `backend/app/db.py`

현재 retrieval 구조:

1. `embed_query(query)` 로 Vertex embedding 생성
2. `LawChunk.embedding.cosine_distance(query_vector)` 로 pgvector 검색
3. `law_chunks` 테이블에서 top-k 조회

즉 after 검색은 아래 3가지를 전제로 한다.

- PostgreSQL
- pgvector
- DB 안에 저장된 768차원 embedding

---

## 4. 현재 로컬 파일로 무엇이 있는가

`backend/data/law_chunks/all_chunks.json` 확인 결과:

- 길이: `1722`
- 포함 필드:
  - `chunk_id`
  - `citation_label`
  - `content`
  - `content_normalized`
  - `embedding_text`
  - 각종 법령 메타데이터

중요:

- **`embedding` 벡터 필드는 없다.**

즉 이 파일은 “검색 대상 원문 코퍼스”로는 충분하지만,
- 현재와 같은 벡터 유사도 검색을 그대로 재현하는 저장소는 아니다.

---

## 5. after를 DB 없이 바꾸는 방법

### 방법 A. 로컬 lexical retrieval 로 교체

가장 현실적이고 빠른 방법.

아이디어:

- `all_chunks.json` 를 메모리로 로드
- 질의와 `citation_label`, `content_normalized`, `embedding_text` 의 토큰 겹침을 점수화
- 상위 `top_k` 를 반환

장점:

- DB 제거 가능
- 구현 난이도 낮음
- 1722개 규모면 메모리 선로딩도 부담 적음

단점:

- 현재 pgvector retrieval 과 검색 품질이 달라진다
- demo freeze 와 평가 결과가 흔들릴 수 있다

### 방법 B. 로컬 embedding cache 파일 추가

아이디어:

- 한 번 임베딩을 생성해서 JSON/NPY 같은 로컬 파일에 저장
- 서버 시작 시 벡터를 메모리로 읽고 cosine similarity 계산

장점:

- DB 없이도 현재 retrieval 방식과 유사한 구조 유지 가능

단점:

- 새 산출물 파일이 필요
- 임베딩 생성 과정은 여전히 Vertex 또는 별도 임베딩 소스가 필요
- 구현 난이도와 관리 비용이 lexical 방식보다 크다

### 방법 C. 현재 구조 유지 + DB만 제거 시도

불가능에 가깝다.

이유:

- 검색 로직이 SQLAlchemy model + pgvector 연산에 직접 묶여 있다.
- 단순히 `DATABASE_URL`만 제거해서는 동작하지 않는다.

## 6. 현실적인 추천안

현재 시점에서 가장 현실적인 선택은 두 가지다.

### 추천 1. before는 로컬 파일 기반 유지

- before 실제 API를 붙일 때도 DB를 넣지 말고 원본 구조를 유지한다.

### 추천 2. after는 “로컬 fallback retrieval”을 별도 모드로 추가

즉:

- 기본 모드: 기존 DB + pgvector
- fallback 모드: `all_chunks.json` 기반 lexical retrieval

이 방식이 좋은 이유:

- 기존 freeze 구조를 완전히 버리지 않는다.
- DB가 없는 환경에서도 최소 동작이 가능해진다.
- 문서 초안 demo 와 answer path 를 로컬 환경에서 더 쉽게 점검할 수 있다.

---

## 7. after를 local fallback 으로 바꾸려면 손볼 파일

### 필수

- `backend/app/services/retrieval.py`
  - DB 검색 함수와 local 검색 함수를 분리
- `backend/app/db.py`
  - import 시점 강제 실패 구조를 완화하거나 fallback 모드일 때 우회
- `backend/app/services/__init__.py`
  - retrieval import 부작용 최소화 검토

### 신규 추가 추천

- `backend/app/services/local_corpus.py`
  - `backend/data/law_chunks/all_chunks.json` 로드 및 캐시
- `backend/app/services/local_retrieval.py`
  - lexical / token overlap 기반 검색

### 선택

- `backend/.env`
  - 예: `RETRIEVAL_BACKEND=db|local`

## 8. 주의할 점

after 에서 DB만 제거해도 **Vertex AI answer generation 은 별도 문제로 남는다.**

즉:

- retrieval 을 local file 로 바꿔도
- 현재 `answer_generation.py` 는 여전히 Vertex AI 모델 호출을 사용한다.

따라서 DB 제거 후에도 live answer path 에는 여전히 아래가 필요하다.

- `GCP_PROJECT`
- `GCP_LOCATION`
- `GOOGLE_APPLICATION_CREDENTIALS` 또는 ADC

단, retrieval 에서 query embedding 을 더 이상 안 쓰게 되면
- Vertex 의존성은 “검색 + 답변”에서 “답변 생성만”으로 줄어든다.

---

## 9. 최종 판단

### before

- DB 없이 로컬 파일 기반으로 충분히 가능
- 원래도 그렇게 설계되어 있음

### after

- 현재 구조 그대로는 DB 제거 불가
- 다만 검색 방식을 바꾸면 로컬 파일 기반으로 충분히 전환 가능
- 가장 현실적인 방법은 `all_chunks.json` 기반 local lexical fallback 추가
