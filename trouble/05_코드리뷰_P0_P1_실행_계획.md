## 범위

이번 실행 계획은 아래 항목만 다룬다.

1. `P0` 업로드 원본/산출물의 무인증 공개 제거
2. `P1` 업로드 크기/개수 제한 추가
3. `P1` fresh environment 부팅 불안정 제거

이번 계획에서 의도적으로 제외하는 항목:

- PDF preview 개선
- `before`/`after` 코퍼스 단일화
- `legend` semantics 복구

위 항목들은 후속 `P2` 패치에서 별도로 다룬다.

## 원칙

- 보안 수정과 데모 확장을 섞지 않는다.
- API 응답 contract는 필요한 최소 범위만 바꾼다.
- `before` UX는 가능하면 유지하되, 민감정보 노출을 전제로 한 UI는 과감히 축소한다.
- backend와 frontend를 한 번에 깨지지 않도록, 응답 변경 전에 프런트 fallback을 먼저 준비하거나 같은 패치 안에서 함께 수정한다.

2. `run_contract_review_pipeline()` 응답에서 `uploaded_files[*].path` 제거

## 변경 순서 이유

이 순서가 필요한 이유는 다음과 같다.

1. 프런트 fallback 없이 backend에서 응답 필드를 바로 제거하면 `/before` 결과 화면이 깨질 수 있다.
2. artifact 제거는 보안상 가장 시급하고, 동시에 부팅 안정성 문제도 일부 같이 해소한다.
3. 업로드 제한은 기능을 막는 변경이므로, 그 전에 현재 동작 경로를 정리해 두는 것이 안전하다.
