# Code Review Risk And Runtime Impact

기준일: `2026-04-21`

공통 전제:

- 모든 수정은 현재 사용자에게 보이는 UI 구조와 기존 기능 동작을 유지하는 범위에서만 진행한다.
- 정리, 보안 보강, 안정화, 리팩터링이 필요하더라도 현재 확인 가능한 화면 흐름과 기능 결과를 바꾸지 않는 것을 우선한다.

대상: 현재 working tree 기준 `before` 통합 패치와 메인 랜딩/폼 접근성 변경

목적:

- 코드리뷰에서 식별한 위험을 위험도별로 다시 정리
- 각 위험을 수정했을 때 전체 코드 실행에 미치는 영향을 사전에 판단
- 수정 우선순위와 검증 포인트를 한 문서에서 확인

## 요약

현재 패치에서 실제 실행 관점의 핵심 위험은 아래 순서로 크다.

1. 업로드 원본/산출물 공개 노출
2. 업로드 입력 상한 부재
3. fresh clone 또는 깨끗한 환경에서의 `before` 서버 부팅 불안정
4. PDF 업로드 경로와 결과 렌더링의 불일치
5. `before` 전용 법령 코퍼스 분기
6. `legend` 제거로 인한 폼 접근성 회귀

실행 영향 기준으로 보면 `1~3`은 운영 안정성 또는 보안에 직접 영향을 주므로 우선 수정해야 한다.  
`4~6`은 서비스 전체를 즉시 멈추게 하지는 않지만, 대표 사용자 경로와 정합성에 영향을 준다.

  - 배포 환경에서 공개 정적 파일 운영 부담이 줄어든다.
- 주의할 영향:
  - 현재 `ResultPanel`의 업로드 원본 미리보기 UI는 그대로 두면 깨진다.
  - 따라서 프런트는 원본 미리보기를 숨기거나, 새 다운로드/프록시 경로로 교체해야 한다.
- 회귀 가능성:
  - `/before` 결과 화면에서 "업로드 원본 확인" 카드가 사라지거나 동작 방식이 바뀐다.
  - 데모 자료가 이 UI를 전제로 되어 있으면 설명 문구 보정이 필요하다.

검증 포인트:

  - PDF 업로드 경로가 대표 데모 경로로 사용 가능해진다.
- 주의할 영향:
  - 프런트 타입과 응답 구조가 조금 늘어날 수 있다.
  - 새 UI 분기에서 이미지/PDF 혼합 처리 실수 가능성이 있다.
- 회귀 가능성:
  - 이미지 미리보기 경로를 건드릴 때 기존 JPG/PNG 렌더링이 같이 깨질 수 있다.

검증 포인트:

- JPG 단일 업로드 후 결과 화면 정상 렌더링
- PDF 단일 업로드 후 결과 화면이 깨지지 않고 적절한 대체 UI 표시

## 수정 우선순위

### 1차 즉시 수정

- 업로드 원본/산출물 공개 제거
- 서버 내부 경로 응답 제거
- 업로드 크기/개수 제한 추가
- `before` 부팅 환경 의존성 제거

이 단계의 전체 실행 영향:

- 보안과 운영 안정성이 즉시 좋아진다.
- 대신 `/before` 결과 화면의 원본 확인 카드가 바뀌거나 사라질 수 있다.
- 따라서 백엔드만 고치면 안 되고 프런트도 함께 정리해야 한다.

### 2차 후속 수정

- PDF 결과 렌더링 정합화
- `before`/`after` 법령 코퍼스 단일화
- `legend` semantics 복구

이 단계의 전체 실행 영향:

- 기능/정합성/a11y 완성도가 올라간다.
- 일부 before 결과 문구와 데모 화면이 바뀔 수 있으므로 스모크와 문서 업데이트가 필요하다.

## 결론

이번 변경에서 가장 큰 문제는 "before 기능 추가" 자체가 아니라, 그 기능이 현재 저장소의 보안 원칙과 운영 재현성 기준을 어긴 상태로 연결돼 있다는 점이다.

따라서 전체 코드 실행 안정성 관점에서는 아래 순서로 처리하는 것이 맞다.

1. 공개 artifact 제거와 응답 최소화
2. 입력 제한으로 메모리/비용 방어
3. fresh environment 부팅 보장
